# DeepSeek API Documentation (Consolidated)

This document consolidates the DeepSeek API documentation available at https://api‑docs.deepseek.com/. It combines the reference, guides, pricing, FAQs and news announcements into a single machine‑readable Markdown file. Each section includes references to the original documentation lines for verification.

Overview
The DeepSeek API provides programmatic access to large language models for chat completions, reasoning and fill‑in‑the‑middle tasks. Models such as deepseek-chat and deepseek-reasoner can be called via HTTP endpoints. Authentication uses an API key sent in the HTTP Authorization header. The API includes endpoints for creating chat completions, retrieving models and checking user balance. The API is stateless; to conduct multi‑round conversations the client must supply the entire conversation history.

Streaming responses typically reduce latency compared with non‑streaming requests

API Endpoints
Authentication
All endpoints require an API key. Include the key in the HTTP header:

http
Copy
Authorization: Bearer {YOUR_API_KEY}
Common Parameters
model: required; choose deepseek-chat or deepseek-reasoner for chat completion; deepseek or similar for FIM tasks.

temperature: controls output randomness. Recommended values: coding/math tasks 0.0, data analysis 1.0, general conversation 1.3, translation 1.3, creative writing 1.5
api-docs.deepseek.com
.

top_p: nucleus sampling parameter (0.0–1.0). Default 1.0.

max_tokens: maximum output tokens; default 512; must be ≤ model’s max output tokens.

stop: list of strings to end generation.

stream: when true, the API returns a Server‑Sent Events stream, enabling incremental output.

POST /chat/completions – Create Chat Completion
This endpoint generates chat responses with multi‑turn memory and optional tool/function calls
api-docs.deepseek.com
.

Request body
api-docs.deepseek.com
:

json
Copy
{
  "model": "deepseek-chat" | "deepseek-reasoner",
  "messages": [
    {"role": "system" | "user" | "assistant", "content": string}
    ...
  ],
  "stop": ["string", ...],
  "stream": boolean,
  "temperature": number,
  "top_p": number,
  "max_tokens": integer,
  "n": integer,
  "logprobs": integer,
  "presence_penalty": number,
  "frequency_penalty": number,
  "user": string,
  "tools": [
    {
      "type": "function",
      "function": {
        "name": string,
        "description": string,
        "parameters": {/* JSON Schema defining arguments */}
      }
    }
  ],
  "tool_choice": "none" | "auto" | {"type": "function", "function": {"name": string}},
  "response_format": {"type": "text" | "json_object"},
  "cache": boolean
}
Fields:

messages: required list containing previous conversation messages. Each message has a role (system, user, assistant, tool) and content. Tools produce messages with role tool and a tool_call_id parameter
api-docs.deepseek.com
. Chat history must be provided since the API is stateless
api-docs.deepseek.com
.

tools & tool_choice: define functions the model can call. Each tool includes a JSON Schema describing parameters
api-docs.deepseek.com
. tool_choice can force a specific tool or disable tool invocation. When a function is called, the response’s choices[].message will include a tool_calls array listing calls with id, function.name and function.arguments. After executing the function, you must append another message with role tool and content equal to the tool’s result before continuing
api-docs.deepseek.com
.

response_format: set to { "type": "json_object" } to enforce valid JSON output. Must also ask the model to provide JSON in the prompt
api-docs.deepseek.com
.

cache: boolean to enable or disable context caching. When true (default), repeated prefixes are cached and billed at a reduced price
api-docs.deepseek.com
.

Response
api-docs.deepseek.com
:

json
Copy
{
  "id": string,
  "model": string,
  "created": unixTimestamp,
  "choices": [
    {
      "index": integer,
      "message": {
        "role": "assistant" | "tool",
        "content": string or null,
        "tool_calls": [
          {"id": string, "type": "function", "function": {"name": string, "arguments": string}}
        ]
      },
      "finish_reason": "stop" | "length" | "tool_calls" | "content_filter"
    }
  ],
  "usage": {
    "prompt_tokens": integer,
    "completion_tokens": integer,
    "total_tokens": integer,
    "cache_hit_tokens": integer
  }
}
finish_reason indicates why generation stopped. length occurs when max_tokens is reached; tool_calls indicates a function call; content_filter means the output was filtered for safety
api-docs.deepseek.com
.

usage.cache_hit_tokens records tokens retrieved from cache
api-docs.deepseek.com
.

Streaming: When stream=true, the API returns an SSE stream; each chunk is a JSON object preceded by data:. The final line is [DONE]
api-docs.deepseek.com
.

Reasoning Model (deepseek-reasoner) Guide
deepseek-reasoner is a Chain‑of‑Thought (CoT) reasoning model with long context (128k tokens). To enable reasoning mode:

Set model to deepseek-reasoner.

Add a system message instructing the model to think step by step and to include reasoning (“scratchpad”) and final answer.

The model only supports: temperature, top_p, max_tokens, stop, n, stream, user
api-docs.deepseek.com
. It does not support tools, logprobs, presence/frequency penalties or JSON output
api-docs.deepseek.com
.

Example request shows messages with system and user prompts and receives a response containing reasoning_content (the chain‑of‑thought) in addition to the final answer
api-docs.deepseek.com
.

For multi‑turn reasoning, supply the full conversation history and set cache=true (default) to reuse context
api-docs.deepseek.com
.

Multi‑Round Conversation
DeepSeek API is stateless; therefore you must provide the entire conversation history on each call. The typical pattern for multi‑round chat is:

Maintain an array messages with objects {role: "user"|"assistant"|"system", content: string}.

Append new user or assistant messages to the array.

Pass the messages array in each API call
api-docs.deepseek.com
.

This ensures context remains across turns. Without sending history the model will treat each call as independent.

Chat Prefix Completion (Beta)
This Beta feature forces the model to continue code from a specified prefix. Requirements:

Use the Beta API base URL.

The last message’s role must be assistant with prefix: true. This instructs the model to treat the content as a prefix and continue it
api-docs.deepseek.com
.

response_format must remain default (text) because JSON output is not supported in Beta features
api-docs.deepseek.com
.

Example: to generate Python code, you send a system prompt instructing the model to output Python code only. The assistant message includes the prefix def fib(n):\n with prefix: true. The model will return the remainder of the code after the prefix
api-docs.deepseek.com
.

Fill‑in‑the‑Middle (FIM) Completion (Beta)
Fill‑in‑the‑middle completion fills missing code between a prefix and suffix. This is part of the Beta API and uses the /completions endpoint described above. Key points:

Use Beta base URL.

Maximum 4,000 tokens (prompt + suffix + completion)
api-docs.deepseek.com
.

Not supported: tools, tool_choice, response_format=json_object
api-docs.deepseek.com
.

Example: provide prompt with the code prefix, suffix with code after the missing part, and call the API. The response will contain the missing code to insert
api-docs.deepseek.com
.

JSON Output
You can instruct DeepSeek to output structured JSON. Two steps are required
api-docs.deepseek.com
:

Include response_format: { "type": "json_object" } in the request.

Ask the model in your prompt to output a valid JSON (e.g., "Respond in JSON format").

If only the response_format is set and the prompt doesn’t ask for JSON, the model may ignore the format
api-docs.deepseek.com
. Conversely, if you ask for JSON but do not set response_format, the model may output JSON but is not guaranteed. When both conditions are satisfied, the response will always be valid JSON.

Function Calling
DeepSeek supports function calling similar to OpenAI. To use it:

Define functions in the tools array. Each function has a name, description and a parameters object following JSON Schema
api-docs.deepseek.com
.

Optionally set tool_choice to auto (default) so the model decides when to call functions, or specify a particular function to enforce calling
api-docs.deepseek.com
. To disable calls, set tool_choice to none.

In the assistant response, the tool_calls field lists calls with function.name and a JSON arguments string
api-docs.deepseek.com
.

After receiving a tool call, execute the function on the client side and then append a message with role: "tool" and content equal to the function’s result. Continue the conversation by calling the API again
api-docs.deepseek.com
.

Example function definition and call: define a get_weather function with parameters location and unit. When the user asks about weather, the model may respond with a tool call. The client then executes the function, sends the result back and obtains the final answer.
api-docs.deepseek.com

Context Caching
DeepSeek uses context caching to reduce cost and latency when identical prefixes are reused across requests. Features:

Enabled by default – set cache=false to disable
api-docs.deepseek.com
.

Caches only repeated prefixes; new tokens or suffixes are still billed fully
api-docs.deepseek.com
.

Useful for Q&A with long articles, multi‑turn conversations and few‑shot learning. When using caches, repeated tokens are retrieved from disk with near‑zero latency
api-docs.deepseek.com
.

Response includes usage.cache_hit_tokens – number of cached tokens
api-docs.deepseek.com
.

Each unit of cache comprises 64 tokens; caching is probabilistic for units with random sampling
api-docs.deepseek.com
.

Appendices
Tips for Effective Usage
Temperature & Top P: Adjust temperature for creativity vs determinism. For tasks requiring deterministic outputs (coding, math), use low values (0–0.2). For creative tasks (story writing), higher values (1.0–1.5) produce varied outputs
api-docs.deepseek.com
.

System Messages: Start conversations with a system message to set the assistant’s behavior (e.g., language, role, style). When using deepseek-reasoner, instruct step‑by‑step reasoning
api-docs.deepseek.com
.

Caching: Keep repeated prefixes (e.g., long context) consistent to maximize cache hits. Caches are at the level of 64 tokens; small changes may reduce cache effectiveness
api-docs.deepseek.com
.

Handling Keep‑alive Comments: When streaming, remove empty lines or : . lines from the output; they are keep‑alive markers to prevent timeouts
api-docs.deepseek.com
.

Glossary
Prompt tokens: tokens from the input messages (system/user/assistant/tool) provided in the request.

Completion tokens: tokens generated by the model in the response.

Chain‑of‑Thought (CoT): the model’s step‑by‑step reasoning process; accessible via deepseek-reasoner and reasoning_content field.

Server‑Sent Events (SSE): streaming protocol where the server pushes incremental events over a single HTTP connection.

Beta API: alternative base URL enabling experimental features like FIM and chat prefix completion. Features may change or be removed.

Cache hit tokens: number of tokens retrieved from cache (billed at reduced price). Recorded in the response usage fields
api-docs.deepseek.com
.
